# API and Data Model

This document outlines the core data structures utilized across the pipeline. These schemas serve as the contract between the Field Detector, Test Generator, Test Execution, and Report Generator modules.

All models are defined using **Pydantic** for strong typing and validation.

## Snowflake-Ready Data Strategy
The data models are designed to be **normalized** and **flat-compatible**. While the local application uses nested JSON for convenience, the schema structure allows for direct mapping to a relational Snowflake schema without complex transformation logic.

---

## 1. FieldSchema (Detected Metadata)
Represents a single input element extracted from the DOM, serving as the deterministic foundation. Maps to the `DIM_FIELDS` table.

```python
from pydantic import BaseModel
from typing import Optional, List

class FieldSchema(BaseModel):
    field_id: str                # Unique internal ID (e.g., "field_001")
    run_id: str                  # Foreign key to RunSummary
    label: Optional[str] = None  # Extracted associated label text
    name: str                    # The element's name attribute
    id: Optional[str] = None     # The element's DOM ID
    type: str                    # Input type (text, email, number, textarea, etc.)
    required: bool = False       # Is the field mandatory?
    min_length: Optional[int] = None # Explicit minimum length from DOM
    max_length: Optional[int] = None # Explicit maximum length from DOM
    pattern: Optional[str] = None    # Regex pattern attribute from DOM
    min_val: Optional[float] = None  # Minimum value from DOM
    max_val: Optional[float] = None  # Maximum value from DOM

## 2. InferredConstraintSchema (AI Hypothesis)
Represents constraints inferred by the AI Inference module. These are validated deterministically.

```python
class InferredConstraintSchema(BaseModel):
    constraint_id: str           # Unique identifier
    field_id: str                # Foreign key to FieldSchema
    semantic_type: str           # Inferred meaning (e.g., "ssn", "phone", "first_name")
    likely_format: str           # Expected format (e.g., regex or descriptive string)
    confidence_score: float      # AI confidence (0.0 to 1.0)
    source: str                  # e.g., "label_inference", "placeholder_inference", "learned_pattern"
```

## 3. TestCaseSchema (Generated Data)
Represents a single executable input scenario for a specific field generated by the Combination Planner. Maps to the `DIM_TEST_CASES` table.

```python
from pydantic import BaseModel
from typing import Union

class TestCaseSchema(BaseModel):
    test_case_id: str            # Unique test identifier (e.g., "tc_102")
    field_id: str                # Reference to the FieldSchema
    run_id: str                  # Foreign key to RunSummary
    input_value: Union[str, int, float, bool] # The actual payload to inject
    category: str                # e.g., "boundary-max", "fuzz", "valid", "negative"
    expected_outcome: str        # "accept" or "reject"
```

## 4. ResultSchema (Observed Outcome)
Represents the outcome of a submitted test case. Maps to the `FACT_TEST_RESULTS` table.

```python
from pydantic import BaseModel
from typing import Optional

class ResultSchema(BaseModel):
    test_case_id: str            # Reference to the TestCaseSchema
    run_id: str                  # Foreign key to RunSummary
    observed_outcome: str        # "accept", "reject", "error", or "timeout"
    status: str                  # "pass" (observed == expected) or "fail" (observed != expected)
    validation_message: Optional[str] = None # Text of any error message shown on screen
    screenshot_path: Optional[str] = None    # Local path to failure evidence
    execution_duration_ms: int   # Time taken to run the case
```

## 5. RunSummarySchema (Final Report)
Represents the aggregated results of an entire form evaluation session. Maps to the `FACT_RUN_SUMMARIES` table.

```python
from pydantic import BaseModel
from typing import List
from datetime import datetime

class FieldMetrics(BaseModel):
    total_fields_detected: int
    total_tests_executed: int
    total_passed: int
    total_failed: int
    pass_rate_percentage: float

class RunSummarySchema(BaseModel):
    run_id: str                  # Unique session ID
    timestamp: datetime          # Execution time
    target_url: str              # The URL under test
    metrics: FieldMetrics
```

---

## Normalized Export Model (Snowflake Target)
When exporting for Snowflake (Phase 3), the system will produce a set of flattened JSON or CSV files corresponding to the following relational structure:

1.  **FACT_RUN_SUMMARIES**: `run_id`, `timestamp`, `target_url`, `total_fields`, `total_tests`, `pass_rate`.
2.  **DIM_FIELDS**: `field_id`, `run_id`, `name`, `type`, `required`, etc.
3.  **DIM_INFERRED_CONSTRAINTS**: `constraint_id`, `field_id`, `semantic_type`, `likely_format`, `confidence_score`.
4.  **FACT_TEST_RESULTS**: `test_case_id`, `run_id`, `observed_outcome`, `status`, `duration_ms`, `validation_message`.

This normalization ensures that historical trends (e.g., "failure rate of email fields over time") can be queried efficiently using standard SQL in Snowflake.
